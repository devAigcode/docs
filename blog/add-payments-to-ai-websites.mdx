## Why Login Comes Before Payment

In a real business, payment usually needs to be tied to user identity. Who owns the order, who can enter the backend, and which accounts have permission to manage the project all require support from an identity system.

So I configured Google signup first. After opening Google sign-in, the system provided two options: one was to let AutoCoder manage the OAuth configuration, and the other was to use my own Google Cloud OAuth credentials.

<Frame caption="Configuring Google sign-in: let AutoCoder manage OAuth, or use your own Google Cloud credentials.">
  ![Google signup configuration screen offering managed OAuth or custom Google Cloud OAuth credentials](/images/blog/add-payments-to-ai-websites/03.png "Google sign-in OAuth configuration options")
</Frame>

After the configuration was completed, AutoCoder returned a specific result: the backend login page now had a Sign in with Google button, the Google callback would be handled on the login page, and only linked administrator emails could log in to the backend.

<Frame caption="AutoCoder confirms Google sign-in is live on the admin login, with callbacks and access control handled.">
  ![Confirmation panel showing Google email sign-in added to the admin login page with access control notes](/images/blog/add-payments-to-ai-websites/04.png "Google admin sign-in enabled confirmation")
</Frame>

This shows that it handled more than just a frontend button. It also involved login callbacks, access control, and backend redirection. For the later payment setup, this was equivalent to adding the identity foundation first.

## The Stripe Payment Foundation Has Been Set Up, but Checkout Still Needs Business Pricing

In this test, I chose Stripe. AutoCoder then returned the result: Stripe payment setup has been started.

It had already completed several things: setting Stripe as the payment provider, adding a purchase order structure, and confirming that the existing storefront already had products that could be used for a future checkout flow.

<Frame caption="AutoCoder reports that Stripe payment setup has started and a purchase order structure was added.">
  ![Status panel stating Stripe payment setup has been started, listing completed, blocked, and ready items](/images/blog/add-payments-to-ai-websites/07.png "Stripe payment setup started status")

At the same time, the system also explained the foundational capabilities that were now prepared: the purchase order layer supports statuses such as pending, paid, canceled, and failed, and Stripe session tracking had been added.

But the most important point here is that AutoCoder did not directly claim that payment had been fully completed. It clearly pointed out that one key piece of business information was still missing: the products did not yet have a confirmed online pricing model.

<Frame caption="The pricing decision AutoCoder surfaces before a checkout flow can be finalized safely.">

For a jewelry website, this judgment is very realistic. High-end jewelry may have fixed prices, may use price on request, or may only allow certain products to be purchased online. Without confirmed pricing, the system cannot safely generate the final checkout flow.

This step does not represent failure. Instead, it is closer to how real business development works: payment functionality requires both technical configuration and business rules.

## Hosted Checkout: Payment Information Is Completed in a Secure Window

After clicking purchase, the page enters the payment confirmation flow. In the screenshot, the right side of the website keeps the order summary, including the product name, size, quantity, price, shipping fee, and total amount.

At the same time, the system opens a hosted checkout window provided by Clink. Users need to fill in their email, card information, cardholder name, and billing address in this window.

<Frame caption="The Clink hosted checkout window opens beside the live order summary for a secure payment.">
  ![Clink hosted checkout window collecting email, card details, and billing address next to an order summary totaling 8,600](/images/blog/add-payments-to-ai-websites/11.png "Clink hosted checkout window")
</Frame>

The key point here is that payment information is not casually embedded into a normal page. Instead, it is handled through hosted checkout. For payment scenarios, this method is better suited to reducing the security burden on both users and website developers.

At the bottom of the page, the system also displays the following prompt:

> Hosted payment checkout will open in a secure window.
>
> Payment confirmation is in progress. Please keep this page available.

This shows that the payment flow is not a static form. It is a complete chain that includes the payment window, payment confirmation, and status return.

## Test Environment: Use Test Cards to Verify the Payment Flow

In the preview environment, AutoCoder.cc clearly states: **All payments in preview are in test mode.**

In other words, no real charge will be made here. Users need to use test card numbers to verify whether the payment flow can run successfully.

The prompt on the left provides test card information, for example:

- **4242 4242 4242 4242** is used to simulate a successful payment;
- **4000 0000 0000 3220** is used to simulate 3D Secure authentication;
- **4000 0000 0000 0002** is used to simulate a failed payment.

<Frame caption="The checkout page with shipping and payment methods, and Stripe test cards listed in the project sidebar.">
  ![Aura Jewelry checkout showing shipping address and payment method options with test cards displayed in the sidebar](/images/blog/add-payments-to-ai-websites/12.png "Checkout page with test card guidance")

This step is very important for a real business.

A payment feature cannot only verify the "successful payment" scenario. It also needs to test different states such as 3D Secure, payment failure, payment cancellation, and confirmation callbacks. Only when these states can all be handled correctly does the website have the foundation for launching payment functionality.

Here, AutoCoder.cc directly displays the test cards in the project sidebar. Users can follow the prompt and enter the test card number into the payment window to complete verification.
